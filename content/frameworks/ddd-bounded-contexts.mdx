## The Algorithm

1. **Найдите subdomain**: Какую бизнес-область покрывает сервис — одну или несколько?
2. **Выделите aggregates**: Группы объектов с invariant, изменяемые как единое целое.
3. **Нарисуйте границы контекста**: Где модель «заканчивается» — по языку, не по repo.
4. **Разделите domain и infrastructure**: Repositories, controllers — infrastructure; правила расчёта — domain.
5. **Отметьте anemic domains**: Логика в services вместо entities — сигнал размытых границ.

## Anti-patterns

- **BC = microservice**: Контекст — semantic boundary, deploy unit может быть другим.
- **Anemic domain model**: Entities без поведения — DDD на бумаге, transaction script в коде.
- **Big ball of mud как один BC**: «Весь монолит — один контекст» обычно означает «контекст не определён».
