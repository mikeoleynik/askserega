## The Algorithm

1. **Определите границу (Seam)**: Найдите точку, где можно перехватить вызовы — на уровне API Gateway, шины сообщений или feature flag.
2. **Настройте маршрутизацию (Facade)**: Создайте слой проксирования, который будет направлять часть трафика в новый сервис, а остальное — в монолит.
3. **Выделите первый срез (Vertical Slice)**: Выберите один эндпоинт или одно событие — минимальный, но сквозной бизнес-сценарий.
4. **Итерируйте**: Переносите функциональность шаг за шагом: старый код уменьшается, новый сервис разрастается.
5. **Выведите legacy из эксплуатации**: Когда трафик упадет до нуля, а данные будут полностью мигрированы — удалите старый код.

## Anti-patterns

- **«Большой взрыв» (Big Bang)**: Попытка вырезать все функции сразу за один спринт. Полностью перечеркивает саму идею постепенной и безопасной миграции.
- **Замена без маршрутизации**: Вызов нового сервиса напрямую из кода монолита вместо использования прокси/шлюза. Лишает возможности мгновенно откатить трафик назад.
- **Игнорирование миграции данных**: Перенос эндпоинтов без четкого плана разделения и синхронизации баз данных. Гарантированно приводит к эффекту расщепления сознания (Split-Brain).
