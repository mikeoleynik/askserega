## The Algorithm

1. **Инвентаризация команд**: Определите, кто и чем владеет — по штатному расписанию и по факту (через git blame, дежурства и логи коммитов).
2. **Инвентаризация границ системы**: Выделите модули, микросервисы, изолированные контексты (Bounded Contexts) и независимые единицы развертывания.
3. **Наложение карт**: Сопоставьте структуры. Идеальный фокус: одна команда — один контекст, один сервис — одна команда.
4. **Поиск точек трения**: Найдите места, где для одного релиза или изменения требуется синхронизация трех и более команд.
5. **Планирование выравнивания**: Спроектируйте целевое состояние (разделение команды, распил сервиса или выделение платформенной команды) с учетом компромиссов.

## Anti-patterns

- **Закон Конвея как оправдание плохой архитектуры**: «Так сложилось» — не стратегия; карта показывает куда двигаться.
- **Игнорирование платформенных команд**: Попытка сделать абсолютно все команды продуктовыми. Без выделения платформенных и поддерживающих команд (инфраструктура, общие инструменты) продуктовые команды быстро увязнут в рутине.
- **Conway Mapping без Context Mapping**: Проводить границы только по репозиториям или должностям, забывая про логику бизнеса. Без карты ограниченных контекстов (DDD) вы получите ложную картину и не увидите реальных конфликтов в коде.
