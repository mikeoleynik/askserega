---
title: Sequence Diagrams
subtitle: Обмен сообщениями во времени
difficulty: low
domain_layer: Behavioral
intent: |
  Визуализировать последовательность взаимодействия участников системы в рамках одного сценария. Фреймворк помогает детально спроектировать или задокументировать логику работы процесса, определяя точный порядок обмена вызовами, API-запросами и событиями во времени.
prompt: |
  Спроектируй Sequence Diagram для асинхронного сценария: "[Название сценария]".

  1. Участники (Укажи типы):
    - Сервисы (Инициаторы, Продюсеры, Консьюмеры)
    - Инфраструктура (Брокеры сообщений, Очереди/Топики, БД)
    - Внешние системы (API)

  2. Логика взаимодействия:
    - Синхронные вызовы (HTTP/gRPC) обозначай прямыми стрелками `->`, ответы — пунктиром `-->`.
    - Асинхронную публикацию событий в брокер и их доставку консьюмерам обозначай стрелками `->>`.
    - Явно укажи названия топиков/очередей и структуру ключевых событий.

  3. Граничные случаи:
    - Изолируй альтернативные сценарии (ошибки валидации, таймауты, retry-логику) в блоки `alt` или `opt`.

  Сгенерируй код диаграммы в формате [PlantUML / Mermaid].
theory_anchor: |
  Точный контракт поведения системы: Наглядную схему, где порядок сообщений жестко зафиксирован во времени. Если реальный распределенный трейс в тестовой среде не совпадает с этой диаграммой — это готовый сигнал о том, что архитектурное знание устарело.
artifacts:
  - label: пример Mermaid flowchart
    filename: standard-diagram.md
    open_url: https://mermaid.live/edit
    open_label: Открыть в Mermaid Live
    example: |
      sequenceDiagram
        autonumber
        title Сценарий: [Название процесса]

        actor User as Клиент
        participant API as API Gateway
        participant Broker as Брокер (Топик: orders.v1)
        participant Worker as Сервис обработки
        participant DB as База Данных

        %% Синхронный слой
        User->>API: POST /api/v1/orders (Запрос)
        activate API
        API->>DB: Проверить лимиты
        activate DB
        DB-->>API: Лимиты в норме
        deactivate DB

        %% Асинхронный слой (EDA)
        API-)Broker: Публикация события (OrderCreated)
        API-->>User: HTTP 202 Accepted (Ответ)
        deactivate API

        %% Обработка события консьюмером
        Broker-)Worker: Доставка события (OrderCreated)
        activate Worker

        alt Успешный сценарий
            Worker->>DB: Изменить статус заказа
            activate DB
            DB-->>Worker: Успешно
            deactivate DB
        else Ошибка / Таймаут
            Worker->>Worker: Логирование / Retry-логика
        end

        deactivate Worker
summary: Инструмент для детального моделирования одного сценария работы системы. Отображает точный порядок взаимодействия компонентов (вызовы, API, события) с привязкой к шкале времени.
algorithm:
  - title: Возьмите один сценарий из Runtime Flow Mapping
    description: Не проектируйте с нуля — детализируйте конкретный цепочечный список переходов (hop list).
  - title: Определите линии жизни (lifelines)
    description: Вынесите каждого участника в отдельную колонку (сервис, база данных, брокер, внешнее API).
  - title: Отрисуйте основной успешный сценарий (happy path)
    description: Синхронные вызовы обозначайте сплошными стрелками, асинхронные сообщения — открытыми (пунктирными или стрелками без заливки).
  - title: Добавьте блоки активности
    description: Выделите периоды времени, когда компонент сфокусирован на обработке конкретного вызова.
  - title: Проверьте схему по реальному трейсу
    description: Сверьте получившуюся диаграмму с распределенным трейсом (distributed trace) в тестовой среде (staging).
antipatterns:
  - title: Диаграмма на 30 участников
    description: Перегружает схему. Разбивайте процесс на отдельные поддиаграммы (sub-diagrams) в рамках своих ограниченных контекстов (bounded contexts).
  - title: Детализация до каждого SQL-запроса
    description: 'Превращает схему в нечитаемый лог. Группируйте вызовы: пишите «Сохранить заказ» вместо перечисления 15 мелких INSERT/UPDATE.'
  - title: Игнорирование оси времени
    description: Проектирование шагов без жесткой хронологии. Диаграмма последовательности теряет смысл, если нарушен строгий порядок выполнения сверху вниз.
