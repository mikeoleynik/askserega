subtitle: Проверка контрактов между сервисами
difficulty: medium
domain_layer: Runtime

  - label: A Guide to Contract Testing

  - label: Contract testing (dev.to)
    url: https://dev.to/keploy/what-is-contract-testing-a-knowledge-guide-1l10
intent: |
  Гарантировать, что вызывающая сторона (consumer) и поставщик сервиса (provider) совместимы по API, и автоматически выявлять ломающие изменения (breaking changes) до деплоя в продакшен, не запуская ресурсоемкие сквозные (end-to-end) тесты.
prompt: |
  Спроектируй Contract Testing для интеграции [Название Consumer-сервиса] ↔ [Название Provider-сервиса].

  Выполни задачу по шагам:

  1. Выбери самую хрупкую и критичную точку интеграции (integration point) между ними для первого контракта и обоснуй выбор.
  2. Определи формат и схему контракта (REST/OpenAPI, JSON Schema, gRPC proto или AsyncAPI для event-driven).
  3. Опиши Consumer-driven подход:
    - Сформулируй ожидания (expectations) на стороне Consumer (какие поля/статусы критичны, какие моки генерировать).
    - Опиши схему верификации на стороне Provider (как запускать тесты на реальном коде поставщика).
  4. Напиши пример спецификации контракта (в коде или JSON/YAML) и шаблон теста для Consumer и Provider на [стек/язык, например: Go/Java/TS/Pact].

  Результат должен быть готов к переносу в репозиторий.
theory_anchor: |
  Прозрачную и безопасную интеграцию сервисов. Контрактное тестирование переводит абстрактные договоренности между командами в явный код. Вы получаете автоматический контроль совместимости: любое ломающее изменение на стороне поставщика сервиса гарантированно валит сборку в CI, исключая неожиданные сбои на продакшене.
artifacts:
  - label: Файлы контрактов (Пример JSON-схемы запроса/ответа)
    filename: contract.json
    example: |
      {
        "$schema": "http://json-schema.org/draft-07/schema#",
        "title": "UserIntegrationContract",
        "type": "object",
        "properties": {
          "request": {
            "type": "object",
            "properties": {
              "user_id": { "type": "string", "format": "uuid" }
            },
            "required": ["user_id"]
          },
          "response": {
            "type": "object",
            "properties": {
              "id": { "type": "string", "format": "uuid" },
              "email": { "type": "string", "format": "email" },
              "status": { "type": "string", "enum": ["active", "suspended"] }
            },
            "required": ["id", "email", "status"]
          }
        }
      }
  - label: Шаблон конфигурации CI (Пайплайн проверки на GitHub Actions)
    filename: github-actions-demo.yml
    example: |
      name: Contract Verification

      on:
        pull_request:
          branches: [ main ]

      jobs:
        contract-test:
          runs-on: ubuntu-latest
          steps:
            - name: Checkout code
              uses: actions/checkout@v4

            - name: Set up Environment
              uses: actions/setup-node@v4 # или ваш стек (Go, Java, Python)
              with:
                node-version: '20'

            - name: Install dependencies
              run: npm ci

            - name: Run Provider Verification
              run: npm run test:contracts:verify
              env:
                CONTRACT_BROKER_URL: ${{ secrets.CONTRACT_BROKER_URL }}
                CONTRACT_BROKER_TOKEN: ${{ secrets.CONTRACT_BROKER_TOKEN }}
  - label: Матрица покрытия интеграции (Реестр контрактов)
    filename: registry.md
    format: markdown
    example: |
      | Вызывающий сервис (Consumer) | Поставщик (Provider) | Точка интеграции (Эндпоинт / Событие) | Тип контракта (Pact / OpenAPI / Proto) | Статус покрытия |
      | --- | --- | --- | --- | --- |
      | billing-service | user-service | GET /v1/users/id | OpenAPI / JSON Schema | Покрыто |
      | notification-service | order-service | event: order_created | AsyncAPI / Avro | В разработке |
summary: Гарантирует совместимость сервисов по API без запуска тяжелых сквозных тестов. Ломающие изменения отлавливаются до деплоя, делая процесс обновления предсказуемым и безопасным.
algorithm:
  - title: Найти критичные точки интеграции
    description: Выбрать топ-5 самых хрупких связей на основе карты зависимостей (Dependency Mapping) и анализа трафика (Runtime Flow).
  - title: Выбрать формат
    description: Определить инструмент — Pact (подход consumer-driven), сравнение схем OpenAPI (OpenAPI diff) или реестр схем (Schema Registry) для событийной архитектуры.
  - title: Написать первый контракт
    description: Покрыть минимальный успешный сценарий (happy path) и 1–2 основных кейса с ошибками.
  - title: Настроить проверку на стороне поставщика в CI
    description: Автоматизировать верификацию (Provider verification), чтобы сборка поставщика сервиса падала при нарушении контракта.
  - title: Внедрить стратегию версионирования
    description: Определить правила эволюции контрактов, чтобы изменения на бэкенде не ломали текущих потребителей.
antipatterns:
  - title: Использовать контрактные тесты вместо сквозных (E2E)
    description: Контракты защищают только границы сервисов и форматы данных, но не тестируют сквозную бизнес-логику системы.
  - title: Проверки без автоматизации в CI
    description: Если файлы контрактов лежат в репозитории, но не проверяются при каждом пуле — это мертвая документация, которая быстро устареет.
  - title: Один гигантский контракт на всё
    description: Попытка описать всё взаимодействие сервисов в одном файле приводит к путанице. Для каждой точки интеграции (эндпоинта или события) должен быть отдельный изолированный контракт.
