intent: |
  Сформировать общий словарь нефункциональных требований (quality attributes, architectural characteristics, «-ilities») и явно расставить приоритеты для системы. Без этого выбор архитектурного стиля и компромиссы между командами остаются неявными.
prompt: |
  Построй карту quality attributes для системы: [название / контекст].

  1. Собери стейкхолдеров и их ожидания: бизнес, эксплуатация, разработка, безопасность.
  2. Из каталога -ilities выбери 8–12 релевантных атрибутов (не «все подряд»). Для каждого — краткое определение в контексте продукта и измеримый сигнал (метрика, сценарий, SLO).
  3. Ранжируй топ-5 по важности (MoSCoW или веса 1–5). Явно укажи конфликтующие пары (например, availability ↔ cost, modifiability ↔ performance).
  4. Сопоставь приоритеты с известными наборами (FURPS, RASUI, RAMS, agility subset) — что у вас доминирует.
  5. Зафиксируй «анти-цели»: какие -ilities сознательно жертвуем на текущем этапе и почему.

  Результат: таблица приоритетов + 3–5 trade-off, готовых для ADR и выбора архитектурного стиля.
theory_anchor: |
  Архитектурные стили выбирают не «по моде», а по балансу характеристик системы. Каталог quality attributes делает этот баланс обсуждаемым: одни и те же слова (availability, modifiability, reliability) перестают означать разное у разных людей.

  Синхронная связность компонентов «склеивает» их -ilities — поэтому приоритеты атрибутов нужно задавать до детального проектирования интеграций.

    example: |
      **Операционные:** availability, reliability, fault-tolerance, recoverability, resilience, scalability, elasticity.

      **Развитие:** modifiability, maintainability, extensibility, evolvability, testability, deployability.

      **Пользовательские:** usability, learnability, accessibility, performance (responsiveness, timeliness).

      **Интеграция:** interoperability, portability, compatibility.

      **Доверие:** security (confidentiality, integrity, availability), safety, auditability.

      **Известные наборы:** FURPS; RASUI; RAMS; agility = debuggability + extensibility + portability + scalability + securability + testability + understandability.
summary: |
  Каталог нефункциональных характеристик системы (-ilities): availability, modifiability, maintainability, reliability и десятки других. Задаёт общий язык приоритетов и trade-offs перед выбором архитектурного стиля и фиксацией решений в ADR.
algorithm:
  - title: Соберите ожидания стейкхолдеров
    description: Бизнес, ops, dev, security — у каждого свой набор -ilities.
  - title: Отфильтруйте каталог
    description: 8–12 атрибутов, релевантных продукту; не копируйте весь Wikipedia-список.
  - title: Определите измеримость
    description: Для каждого атрибута — метрика, сценарий или SLO, иначе приоритеты не проверить.
  - title: Ранжируйте и найдите конфликты
    description: Топ-5 важных; пары взаимоисключающих целей (availability vs cost).
  - title: Свяжите с решениями
    description: Используйте приоритеты в ADR, выборе стиля (FASA) и fitness functions (Evolutionary Architecture).
antipatterns:
  - title: Все -ilities «высокий приоритет»
    description: Без жертв архитектура не выбирается — явно фиксируйте анти-цели.
  - title: Разные термины в одной команде
    description: «Performance» vs «scalability» vs «responsiveness» — согласуйте глоссарий.
  - title: Каталог без метрик
    description: Список слов не меняет поведение; нужен измеримый сигнал на атрибут.
