  Обеспечить команды единым эталонным шаблоном для проектирования новых и развития существующих сервисов. Reference Architecture устраняет необходимость каждый раз изобретать архитектуру с нуля, закрепляя стандарты по слоям, паттернам и типовым компонентам. Это сокращает время на проектирование и гарантирует, что все системы строятся по общим правилам.

  Сформируй эталонную архитектуру (Reference Architecture) для [название организации / платформы]. Выдай ответ строго в виде Markdown-документа по следующей структуре:

  1. Шаблон эталона:

  2. Анализ соответствия (Трассировка сервиса):

  3. Защитные правила (Fitness Functions):
    Сформулируй 3 жестких, проверяемых правила, которые обязаны соблюдать все новые сервисы (например, требования к изоляции данных, запрет на прямые зависимости, обязательные метрики).

  Пиши лаконично, структурированно, без общих фраз и «воды». Используй устоявшуюся русскоязычную ИТ-терминологию без необозначенного смешения языков.

      ### Архитектурный чек-лист: Запуск нового канала доставки

      Этот чек-лист обязателен к заполнению перед выводом нового сервиса канала доставки (SMS, мессенджеры, push-уведомления) в продакшен. Каждый канал изолируется в отдельном микросервисе для обеспечения отказоустойчивости всей платформы.

      #### 1. Изоляция и границы контекста (Bounded Context)
      - [ ] Сервис отвечает исключительно за доставку через **один конкретный канал** (например, только `delivery-telegram` или только `delivery-whatsapp`).
      - [ ] Внутри сервиса нет сквозной бизнес-логики маршрутизации, каскадных рассылок или выбора приоритета каналов (этим занимается вышестоящий оркестратор).
      - [ ] Сервис имеет собственную изолированную базу данных (или схему в PostgreSQL), другие сервисы не имеют к ней прямого доступа.

      #### 2. Интеграционные контракты и асинхронность
      - [ ] **Входящий поток (Асинхронный):** Сервис принимает сообщения на отправку исключительно через брокер (соответствующую очередь в Kafka/RabbitMQ). Никаких синхронных HTTP-вызовов на отправку со стороны других систем.
      - [ ] **Исходящий поток (Синхронный):** Интеграция с внешним API шлюза/провайдера вынесена в отдельный инфраструктурный адаптер внутри сервиса.
      - [ ] **Обратный поток (Статусы):** Изменение статуса доставки (доставлено, ошибка) публикуется обратно в брокер для обновления общей продуктовой аналитики.

      #### 3. Отказоустойчивость и надежность (Resilience)
      - [ ] Реализован паттерн **Circuit Breaker** на запросы к внешнему API провайдера (если шлюз «лежит», сервис не копит потоки и сразу отвечает ошибкой).
      - [ ] Настроена политика повторных попыток (**Retry Policy**) с экспоненциальной задержкой (Exponential Backoff) для временных ошибок сети (ошибки 5xx от провайдера).
      - [ ] Реализована обработка **Idempotency Key**: повторный прием одного и того же UUID сообщения из очереди не приводит к дублированию отправки пользователю.
      - [ ] Выделена очередь для «плохих» сообщений (**DLQ — Dead Letter Queue**) для складывания некорректных запросов, которые невозможно обработать.

      #### 4. Сквозная функциональность (Cross-cutting Concerns)
      - [ ] **Логирование:** В каждом логе присутствует сквозной идентификатор запроса (`Correlation-ID`), переданный от оркестратора.
      - [ ] **Безопасность:** Секреты и API-ключи внешних провайдеров не хранятся в коде, а внедряются через переменные окружения / Vault.
      - [ ] **Мониторинг:** Настроены и выведены на дашборд метрики:
        - Количество сообщений в очереди (Lag).
        - Процент успешных/ошибочных доставок (Delivery Rate / Error Rate).
        - Время ответа внешнего шлюза (HTTP Latency).

      ### Матрица соответствия сервисов эталонной архитектуре (Compliance Matrix)

      В этой таблице фиксируется текущий статус соответствия микросервисов целевому эталону. Отклонения от эталона допустимы только при наличии утвержденного архитектурного решения (ADR).

      | Идентификатор сервиса | Назначение / Домен | Статус | Выявленные отклонения / Исключения (Exceptions) | Ссылка на ADR | Действие (Action Item) |
      | --- | --- | --- | --- | --- | --- |
      | `delivery-sms` | Канал доставки: SMS-уведомления | **Полное** | Нет. Полностью соответствует эталону изоляции каналов. | — | Не требуется. |
      | `delivery-telegram` | Канал доставки: Telegram-бот | **Полное** | Нет. Реализован по стандартному шаблону. | — | Не требуется. |
      | `delivery-whatsapp` | Канал доставки: Бизнес-чат WhatsApp | **Частичное** | Использует синхронный HTTP Webhook для получения статусов вместо брокера сообщений (ограничение API провайдера). | `docs/adr/012-wa-http-hooks.md` | Принять риски. Пересмотреть контракт при обновлении API провайдера. |
      | `notification-router` | Оркестратор и маршрутизатор уведомлений | **Частичное** | Хранит состояние цепочек отправки в оперативной памяти (Stateful), нарушая требование к Stateless-архитектуре приложения. | `docs/adr/008-router-state.md` | Вынести состояние в Redis до конца Q3 2026. |
      | `legacy-monolith` | Старое ядро системы (Legacy) | **Не соответствует** | Отсутствует декомпозиция на слои, бизнес-логика смешана с базой данных (хранимые процедуры), нет сквозного трейсинга. | `docs/adr/001-legacy-backlog.md` | Поэтапный распил. Вынос каналов доставки в отдельные сервисы по плану миграции. |

      #### Критерии статусов:
      * **Полное:** Сервис соблюдает все 4 блока архитектурного чек-листа.
      * **Частичное:** Есть обоснованные архитектурные отличия, которые зафиксированы в ADR и не ломают общую стабильность системы.
      * **Не соответствует:** Сервис требует рефакторинга или полной переработки, внесен в технический бэклог (обычно относится к legacy-системам).

algorithm:
  - title: Выделите повторяющиеся паттерны
    description: Найдите решения, которые дублируются в 3+ сервисах — это главные кандидаты в эталон.
  - title: Определите слои и границы
    description: 'Четко разграничьте зоны ответственности (например: уровень представления, прикладной слой, бизнес-логика, инфраструктура).'
  - title: Зафиксируйте контракты интеграции
    description: Опишите правила взаимодействия сервисов — синхронный/асинхронный транспорт, авторизация и сквозной мониторинг (observability).
  - title: Зафиксируйте исключения
    description: Явно перечислите допустимые отступления от стандарта и закрепите каждое в формате архитектурного решения (ADR).
  - title: Свяжите с bounded contexts
    description: Определите, какие шаблоны и архитектурные «срезы» применимы к конкретным доменным областям (bounded contexts).
antipatterns:
  - title: Эталон «из воздуха»
    description: Проектирование оторванной от реальности архитектуры. Reference Architecture должна отражать работающую практику, а не превращаться в 100-страничный теоретический талмуд.
  - title: Без процесса обновления
    description: Отсутствие процесса актуализации. Архитектура устаревает, поэтому её необходимо регулярно обновлять через механизм архитектурных решений (ADR) и квартальные ревью.
  - title: Один эталон на microservices и monolith
    description: Попытка применить один эталон и к микросервисам, и к монолиту. Разные архитектурные формы требуют разных моделей проектирования.
