## The Algorithm

1. **Выделите ядро (Domain Core)**: Сформулируйте бизнес-сценарии (Use Cases) и сущности (Entities). Главное правило: ноль импортов сторонних фреймворков, ORM или HTTP-библиотек.
2. **Определите входные порты (Inbound Ports)**: Создайте интерфейсы, через которые внешний мир будет управлять ядром (например, CreateOrderUseCase).
3. **Определите выходные порты (Outbound Ports)**: Создайте интерфейсы для сервисов, которые нужны самому ядру для работы с внешним миром (например, OrderRepository, PaymentGateway).
4. **Реализуйте адаптеры (Adapters)**: Напишите конкретный инфраструктурный код — контроллеры, репозитории, обработчики очередей, — который реализует или вызывает ваши порты.
5. **Проверьте направление зависимостей**: Убедитесь, что все стрелки зависимостей направлены строго внутрь. Инфраструктура зависит от бизнес-логики, но не наоборот.

## Anti-patterns

- **Порты под каждый CRUD-метод**: Не каждый метод репозитория должен быть отдельным портом. Порты создаются под конкретные потребности бизнес-логики, а не просто дублируют функции базы данных.
- **Анемичный гексагон**: Доменное ядро остается пустым, а вся реальная логика оседает в «прикладных сервисах». Архитектура превращается в форму без содержания.
- **Адаптер-божественный класс**: Один гигантский адаптер на тысячи строк кода, который делает всё сразу. Разделяйте адаптеры по логическому смыслу и контекстам.
