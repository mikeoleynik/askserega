prompt: |
  Проанализируй [модуль/класс] с высокой когнитивной и цикломатической сложностью.

  1. Метрики (До): Зафиксируй текущие cyclomatic complexity, lines of code (LOC) и количество зависимостей.
  2. Стратегия: Выбери 3 наиболее эффективных приёма рефакторинга из каталога.
  3. Реализация (для каждого приёма):
    - Обоснуй выбор: какую именно проблему локализует этот шаг.
    - Опиши тесты-страховки: что должно покрывать код перед изменением.
    - Дай пошаговый алгоритм применения (микро-шаги).
    - Покажи контрастный код: до и после (Before / After).
  4. Метрики (После): Продемонстрируй расчет целевых метрик после применения всех 3 шагов.
theory_anchor: |
  Предсказуемый процесс улучшения кода: Переход от хаотичного переписывания к безопасным, контролируемым микро-шагам.

  Инструмент материализации архитектуры: Возможность на практике проверить гипотезу «структура становится проще, поведение остается неизменным».

  Снижение рисков при рефакторинге: Гарантию стабильности системы на каждом этапе за счет жесткой связки «маленький шаг + автоматические тесты».

summary: Каталог паттернов для безопасного изменения структуры кода. Помогает поэтапно снижать сложность — от выделения методов до изменения модульных границ — с помощью мелких контролируемых шагов без риска сломать поведение системы.
algorithm:
  - title: Выявить запах кода
    description: Найти проблемные места (длинный метод, завистливые функции, сквозные изменения) на основе анализа компонентов или архитектурных границ.
  - title: Обеспечить покрытие тестами
    description: Запустить unit-тесты или тесты характеризации — перед началом рефакторинга все тесты должны быть «зелёными».
  - title: Выбрать один паттерн
    description: Найти минимальный шаг для устранения конкретного «запаха» (например, сначала выделение метода, и только затем — выделение класса).
  - title: Применить и проверить
    description: Выполнить шаг рефакторинга, убедиться, что тесты остаются «зелёными», а метрики сложности снизились.
  - title: Повторить
    description: Выстроить цепочку из мелких изменений, фиксируя (commit) каждое успешное состояние кода.
antipatterns:
  - title: Рефакторинг без тестов
    description: Подход «тесты напишем потом» — прямой путь к регрессионным ошибкам.
  - title: Большой взрыв под видом рефакторинга
    description: Попытка переписать всё разом, при которой незаметно меняется поведение системы. Это не рефакторинг.
  - title: Попытка починить самый сложный участок сразу
    description: Начинать нужно с изолированных, покрытых тестами модулей, чтобы сначала накопить уверенность и обкатать шаги.
