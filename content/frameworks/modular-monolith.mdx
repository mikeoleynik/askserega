## The Algorithm

1. **Выделить модули через DDD**: Один ограниченный контекст (Bounded Context) ≈ кандидат на модуль.
2. **Определить публичный интерфейс**: Создать фасад для каждого модуля — это единственная точка входа для остальных компонентов системы.
3. **Зафиксировать правила зависимостей**: Модуль А может вызывать Модуль Б только через его публичный интерфейс; любые внутренние импорты строго запрещены.
4. **Автоматизировать проверку в CI**: Настроить тесты архитектуры (например, ArchUnit или кастомные линтеры), чтобы сборка падала при любом нарушении границ.
5. **Наметить план разделения**: Модуль со стабильным интерфейсом и слабой связностью — первый кандидат на вынос в отдельный микросервис.

## Anti-patterns

- **Модули «на честном слове»**: Создание папок без автоматической проверки границ в CI. Без жестких правил структура каталогов превращается в театр архитектуры.
- **Раздутое «общее ядро» (Shared Kernel)**: Использование общей базы данных или сквозных моделей данных между модулями. Это создает распределенный монолит внутри одной сборки.
- **Монолит как вечная догма**: Стремление сохранить монолитную архитектуру любой ценой. Цель подхода — дать свободу и готовность к безболезненному выделению сервисов, а не законсервировать систему навсегда.
