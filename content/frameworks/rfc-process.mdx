## The Algorithm

1. **Определите threshold**: Когда RFC обязателен — new service, breaking API, data model change, cross-team impact.
2. **Создайте шаблон**: Problem → Proposal → Alternatives → Trade-offs → Rollout → Open Questions.
3. **Назначьте reviewers**: Domain owners, affected teams — до coding.
4. **Проведите review window**: Async comments, sync discussion if needed — зафиксируйте в RFC.
5. **Close with outcome**: Accepted → ADR; Rejected → archive with reasons; Superseded → link to new RFC.

## Anti-patterns

- **RFC после реализации**: Post-facto RFC — theatre, не process.
- **RFC без alternatives**: Proposal без 2+ alternatives — не decision record.
- **RFC graveyard**: Accepted RFCs без ADR follow-up — knowledge теряется на handoff.
